A configuração de anúncios da V1H está em `src/config/configListing.js`. Define listing types, categorias e custom fields. Os custom fields permitem recolher informação específica de cada anúncio (capacidade, comodidades, tipo de espaço) e apresentá-la na interface de forma estruturada.

## Estrutura geral

```
configListing.js exporta:
- listingTypes - array de tipos de listing (com process + unit)
- listingFields - array de custom fields (com schema + filter + show + save config)
- categoryConfiguration - 9 categorias V1H
```

## Listing Types

Cada listing type define o processo de transação associado e as opções de disponibilidade. A combinação de processo e unidade determina como o calendário e os preços funcionam para cada tipo de anúncio.

- **`process`** - qual processo Sharetribe usar (booking, purchase, etc.)
- **`alias`** - versão do processo
- **`unit`** - unidade (day, night, hour, fixed, item, inquiry)
- **`availabilityType`** - `oneSeat` ou `multipleSeats`
- **`stockType`** - `oneItem` / `multipleItems` / `infiniteOneItem` / `infiniteMultipleItems`
- **`transactionFields`** - custom fields da transação

**Exemplos típicos** (configurados/possíveis na V1H):

| listingType | Processo | Unit | Uso |
|---|---|---|---|
| `daily-booking` | default-booking | day | Espaços alugados por dia |
| `nightly-booking` | default-booking | night | Estadias |
| `hourly-booking` | default-booking | hour | Reuniões, eventos curtos |
| `fixed-booking` | default-booking | fixed | Eventos pré-definidos |
| `multi-booking-v10` | default-multi-booking-v10 | hour | Reservas múltiplas com auto-accept |
| `long-term-booking-v10` | default-long-term-booking-v10 | day | Arrendamento mensal/anual |

## Custom Fields - Schema

Cada custom field define onde aparece na interface (filtros, detalhes do listing, wizard de edição), que tipo de dado aceita e quais as validações. A configuração em código deve refletir exatamente o que está definido no Sharetribe Console.

```js
{
  key: 'capacidade',              // ID único
  scope: 'public',                 // public / private / protected
  schemaType: 'long',              // enum / multi-enum / long / boolean / text
  
  // Para enum/multi-enum:
  enumOptions: [
    { option: 'small', label: 'Pequeno' },
    { option: 'medium', label: 'Médio' },
  ],
  
  // Onde aplicar:
  listingTypeConfig: {
    limitToListingTypeIds: true,
    listingTypeIds: ['daily-booking', 'hourly-booking']
  },
  categoryConfig: {
    limitToCategoryIds: true,
    categoryIds: ['trabalho-reunioes']
  },
  
  filterConfig: {
    indexForSearch: true,           // aparece nos filtros de search
    label: 'Capacidade',
    filterType: 'SelectMultipleFilter',
    group: 'primary',                // 'primary' ou 'secondary'
  },
  
  showConfig: {
    label: 'Capacidade',
    isDetail: true                   // mostra na secção Detalhes do listing
  },
  
  saveConfig: {
    label: 'Capacidade do espaço',
    placeholderMessage: 'Selecione...',
    isRequired: true,
    requiredMessage: 'Capacidade obrigatória'
  }
}
```

## Custom Fields típicos V1H

A V1H tem campos extra configurados no Sharetribe Console. Os exemplos abaixo ilustram os tipos de campos mais comuns por categoria.

### Para todas as categorias
- **Comodidades** (`amenities`) - multi-enum (Wi-Fi, Casa de Banho, Acessibilidade, Climatização, Estacionamento)
- **Capacidade** (`numberOfPeople`) - long (número máximo de pessoas)
- **Notas adicionais** (`notes`) - text

### Específicos por categoria
- **Tipo de espaço** (`spaceType`) - enum específico por categoria
- **Equipamento incluído** - multi-enum
- **Restrições** - multi-enum (sem música alta, sem pets, etc.)

## Configuração no Sharetribe Console

Os custom fields devem ser configurados em duas partes independentes. A omissão de qualquer uma delas resulta em comportamento incorreto na interface.

1. **Sharetribe Console** - Listing types + Listing fields (define o schema lá)
2. **`configListing.js`** no código - mostra como renderizar na UI

**Importante**: se mudares no Console, é preciso refletir no `configListing.js` para que a UI saiba como mostrar.

## Como aparece na UI

Cada custom field aparece em até **3 sítios** consoante a sua configuração.

1. **Edit Listing Wizard** (tab Details) - `EditListingDetailsForm`:
   - Renderiza componente apropriado (`FieldSelect`, `FieldCheckbox`, etc.)
   - Aplica validação `saveConfig.isRequired`

2. **Search Page filters** - `FilterComponent`:
   - Se `filterConfig.indexForSearch: true`
   - Aparece em "primary" (sempre visível) ou "secondary" (em "Mais filtros")

3. **Listing Page** - `CustomListingFields`:
   - Se `showConfig.isDetail: true`
   - Mostra na secção "Detalhes" da página

## Listing Page custom fields

Ficheiro: `src/containers/ListingPage/CustomListingFields.js`

Lê os fields configurados, filtra por `listingType` + `categoryLevel1`, e renderiza:
- **enum** - label + valor (ex: "Tipo de espaço: Sala de reuniões")
- **multi-enum** - chips visuais (Wi-Fi / Café / etc.)
- **long** - label + número (ex: "Número de pessoas: 79")
- **boolean** - checkmark ou cross
- **text** - label + texto

**Exclusões**: `averageRating` e `reviewCount` são backend-only mesmo se `isDetail: true`.

## Edição via Edit Listing Wizard

No `EditListingDetailsForm`, cada custom field é renderizado pelo componente `CustomExtendedDataField`:

```js
<CustomExtendedDataField
  fieldConfig={field}
  defaultRequiredMessage={...}
  intl={intl}
/>
```

Que internamente escolhe o componente certo (FieldSelect, FieldCheckboxGroup, FieldNumber, FieldBoolean, FieldTextInput).

## User custom fields

Análogo em `configUser.js` para campos do **utilizador** (não do listing):
- Línguas faladas (já implementado via `LanguagesField`)
- Tipo de utilizador (cliente vs anfitrião vs ambos)
- Outros campos demográficos

Aparece em `ProfileSettingsPage` para edição.

## Indexação para search

Para um campo aparecer nos filtros de search, são necessários dois passos. Ignorar qualquer um deles resulta no campo existir mas não funcionar como filtro.

1. `filterConfig.indexForSearch: true` no `configListing.js`
2. **Sharetribe CLI**: rodar `flex-cli search set` com a config de index

Sem o segundo passo, o campo existe mas não dá para filtrar por ele (busca devolve sempre tudo).

## Exemplos práticos

### Adicionar "Tipo de espaço" como filtro

1. No Sharetribe Console - adicionar enum field `spaceType`
2. No `configListing.js`:
   ```js
   {
     key: 'spaceType',
     scope: 'public',
     schemaType: 'enum',
     enumOptions: [
       { option: 'salon', label: 'Salão' },
       { option: 'studio', label: 'Estúdio' },
       ...
     ],
     filterConfig: {
       indexForSearch: true,
       label: 'Tipo de espaço',
       filterType: 'SelectSingleFilter',
       group: 'primary'
     },
     showConfig: { label: 'Tipo', isDetail: true },
     saveConfig: { label: 'Tipo de espaço', isRequired: true }
   }
   ```
3. Rodar `flex-cli search set` localmente
4. Edit Listing agora pede este campo
5. Search Page mostra filtro
6. Listing Page mostra "Tipo: Salão" nos detalhes
