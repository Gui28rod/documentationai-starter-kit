## StyleguidePage

**URL**: `/styleguide` (não linkado - acesso direto)
**Ficheiro**: `src/containers/StyleguidePage/`

Página oculta de showcase de componentes - útil para devs e designers verem o sistema visual completo sem precisar percorrer o site. Não está linkada na navegação pública.

### O que mostra
- **Cores** da palette V1H (com hex)
- **Tipografia** (Britannic Bold, Century Gothic, Red Hat Text) em vários tamanhos
- **Botões** (Primary, Secondary, InlineText, Disabled, etc.)
- **Form fields** (Text, Number, Checkbox, Select, DatePicker, etc.)
- **Cards** (ListingCard com variantes)
- **Modais** (todos os tipos)
- **Icones** (todos os 40+ ícones SVG)
- **Layouts** (LayoutComposer variantes)
- **Avatars** (Small, Medium, Large)

### Para que serve
- Designers veem o que está disponível antes de pedir novo design
- Devs copiam o exemplo de uso
- QA testa visual em ambos os modos (claro/escuro)

### Como adicionar componente
1. Importar em `StyleguidePage.js`
2. Adicionar secção com `<ComponentExample component={X} />`
3. Definir variantes a mostrar

## Tests

Testes em `src/**/*.test.js` (Jest + React Testing Library). A cobertura é limitada - está focada nos ducks principais e em smoke tests de páginas. Os componentes customizados da V1H têm pouca cobertura de testes automatizados.

### O que está testado

| Componente | Test file |
|---|---|
| `auth.duck` | `src/ducks/auth.test.js` |
| `marketplaceData.duck` | `src/ducks/marketplaceData.test.js` |
| `app` | `src/app.test.js` |
| `EditListingPage` | `src/containers/EditListingPage/EditListingPage.test.js` |
| `SearchPage` | `src/containers/SearchPage/SearchPage.test.js` |
| `ListingCard` | `src/components/ListingCard/ListingCard.test.js` |
| `FieldBoolean`, `FieldSelectTree` | snapshots |

### Cobertura
- **Limitada** - herdada do template Sharetribe
- Focada em ducks principais + smoke tests de páginas
- Pouca cobertura nos componentes V1H customizados (ExtraAlerts, TranslateButton, etc.)

### Como correr
```bash
yarn test                # interactive watch
yarn test --coverage     # com cobertura
yarn test EditListing    # filtrar por nome
```

### Helpers
- `src/util/testData.js` - mocks de listings, users, transactions
- `src/util/testHelpers.js` - render wrappers (Redux store + Router + Intl)

## Analytics

Diretório: `src/analytics/`

### Google Analytics 4

O Google Analytics 4 está configurado mas com gating de consentimento. Só carrega e envia eventos se o utilizador tiver aceitado a categoria "analytics" no banner de cookies.

- `REACT_APP_GOOGLE_ANALYTICS_ID` no `.env`
- Carregado dinamicamente via `includeScripts.js`
- **Gating cookie consent**: só carrega se categoria "analytics" aceita
- Eventos automáticos: page views, scroll depth (via Enhanced Measurement do GA - recomendado **desativar** Enhanced Measurement)

### Plausible (alternativa, não usada)

- `REACT_APP_PLAUSIBLE_DOMAINS` (comentado no `.env`)
- Mais privacy-friendly que GA
- Pago

### Eventos customizados V1H

Triggered via `analytics.duck.js` (ou similar):
- `listing_viewed`
- `favorite_added` / `favorite_removed`
- `search_performed`
- `booking_completed`
- `signup_completed`

## Service Worker

Ficheiro: `public/service-worker.js`

O service worker é um requisito do browser para permitir a instalação da app (PWA). Sem ele, o prompt "Adicionar ao ecrã inicial" não aparece. Atualmente não faz pre-cache de assets porque estes mudam frequentemente em desenvolvimento.

```js
const CACHE_NAME = 'v1hub-pwa-shell-v1';
```

### Handlers
- `install` - cria cache vazio
- `activate` - limpa caches antigos (de versões anteriores)
- `fetch` - **network-first** (passa direto, sem pré-cache)

### Por que sem pré-cache?
Os assets do site mudam frequentemente em dev. Pré-cache levaria a versões desatualizadas. Network-first significa que vai sempre ao servidor primeiro.

### Por que existe se não faz nada útil?
**Requisito para o browser permitir "Install App"**. Sem service worker, o "Add to Home Screen" não aparece.

### Para estender (futuro)
Adicionar:
- Offline page fallback
- Cache de assets estáticos (CSS, fonts)
- Background sync (enviar mensagens quando voltar online)
- Push notifications

## PWA Counter

Ficheiro: `server/data/pwa-counter.json`

Conta quantos utilizadores instalaram a PWA. O contador é incrementado automaticamente após cada instalação bem-sucedida.

### Endpoints
- `GET /api/pwa-counter` - lê
- `POST /api/pwa-counter/increment` - incrementa

## Debugging

### Frontend
- Redux DevTools (extensão browser)
- React DevTools
- Network tab para SDK calls

### Backend
- `console.log` aparece no terminal do servidor
- Sentry (se `REACT_APP_SENTRY_DSN` configurado)
- Logs estruturados em `server/log.js`

### Common debug recipes
- **Listing não aparece na search** - verificar `flex-cli search set` foi corrido para os custom fields
- **Pagamento falha** - ver Stripe Dashboard logs + browser console
- **Notificação não chega** - ver `metadata.unseen*Alerts` do user via Integration SDK
- **Cron não corre** - verificar `DISABLE_*` env vars + logs do servidor
